Recovery Ladder
1. RETRY if transient and safe (with backoff)
2. RESUME FROM CHECKPOINT if interrupted mid-execution
3. REPLAN if output or logic is invalid
4. ESCALATE TO HUMAN if approval or ambiguity blocks progress
5. FAIL TERMINALLY if policy, retry budget, or safety says stop
Idempotency Rule
Read-only tools — retry freely
Mutating tools — retry only if marked idempotent
Otherwise — resume, re-plan, or escalate instead of blind retry
If requires_idempotency=True, RETRY excluded from allowed_actions
Validation Rules
Non-empty policy_id prevents anonymous policies
Unique failure classes no conflicting rules
Non-negative attempts & delays basic safety
max_delay >= base_delay keeps backoff valid
Default unknown rule exists covers unclassified failures
Non-empty decision reason makes recovery auditable
Non-negative retry_after prevents nonsensical wait times
requires_idempotency excludes RETRY side-effect safety
