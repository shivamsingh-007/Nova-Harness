Routing Modes
☰
Rule-based
explicit, debuggable
Default path. Matches task_type, risk_level, and capability_tags against static rules.
☰
Semantic
meaning beyond keywords
Uses request_summary and task_type semantic matching when rules are insufficient.
☰
Hybrid
rules narrow, scores pick
Rules filter candidates, then score-based arbitration picks the winner.
Validation Rules
NON-BLANK routing_id, request_summary, candidate_id, target_id, score_id, constraint_id, decision_id, arbitration_id, routing_id, fallback_id, trigger_reason, envelope_id
CANDIDATES at least one candidate unless BLOCKED status
SELECTION SELECTED/DISPATCHED requires selected_candidate_id
REFERENCE selected_candidate_id must reference an existing candidate
BLOCKED/REJECTED requires decision_reason
CONFIDENCE all scores and confidence bounded 0.0-1.0
ARBITRATION requires at least 2 competing candidates; selection needs tie_break_reason
FALLBACK requires trigger_reason and a RoutingDecisionRecord
HIGH-RISK must_verify constraints honored for high-risk inputs
