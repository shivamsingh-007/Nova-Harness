Verification — 5 Records + 4 Enums
■
VerificationPlan → VerificationResult
Plan is created before execution; Result captures the outcome
VerificationPlan
├─ plan_id: str ● task_id: str ● checks: List[VerificationCheck] (≥1)
VerificationResult
├─ plan_id: str ● run_id: str
├─ checks: List[VerificationCheck]
├─ verdict: VerificationVerdict (PASS/FAIL/PARTIAL/BLOCKED)
└─ started_at: Optional[str] ● finished_at: Optional[str]
Five Records — Key Validators
■
VerificationCheck
check_id, name, type, blocking, command/rule, status, evidence, failure_reason
Must have either command or acceptance_rule
status=FAILED requires failure_reason
Default: blocking=True
■
VerificationEvidence
evidence_id, type (log/report/file/metric), path, summary, metric details
METRIC type requires metric_name + metric_value
Log, report, file: need path or summary
■
VerificationVerdict
status, blocking_failures, non_blocking_failures, summary
Blocking failures + PASS verdict → error
Supports PASS / FAIL / PARTIAL / BLOCKED
■
VerificationPlan + Result
Plan: ≥1 check required. Result: finished_at for PASS/FAIL
Empty plan (no checks) → error
PASS/FAIL without finished_at → error
PARTIAL/BLOCKED: finished_at optional

Recommended Check Order (v1)
1. Targeted test — smallest relevant proof first
2. Lint — code style and basic correctness
3. Typecheck — type safety
4. Acceptance check — matches spec requirements
5. Security (when relevant) — vulnerability scan
6. Artifact presence — output file exists
